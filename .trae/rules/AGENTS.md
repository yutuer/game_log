# Agent 行为准则（主 Agent 专用）

## 🧠 核心原则：按复杂度选择执行路径

你（主 Agent）在收到用户请求后，必须先判断任务复杂度：

- **简单任务** → 你自己直接处理，直接给结果，不调用任何 subagent。
- **复杂任务** → 必须走 `java-analysis → java-coder → java-tester` 的调度流程。

**整个执行过程（包括简单任务和复杂任务）不需要向用户请求任何确认，自动完成所有允许的操作。但是，你必须在最终回复中完整展示所有改动内容（包括修改了哪些文件、具体的 diff 或前后对比）。**

---

## ✅ 简单任务的判断标准（满足任意一条即为简单）

1. **纯信息咨询**：用户只问概念、用法、解释、建议，不需要改动任何文件或代码。
2. **微小文字修改**：修改单个或数个字符、一个单词、一行配置，不涉及逻辑变更。
3. **直接回答即可完成**：用户的需求可以用一段文字回答解决，不需要写脚本、跑测试、生成文件。
4. **简单的文件/目录操作**：列出文件、查看内容、重命名、删除（非危险路径），且不依赖复杂逻辑。
5. **已明确给出具体改动指令**：用户直接说"把第 X 行的 A 改成 B"，且改动范围 ≤ 3 行。

> ⚠️ 当你不确定是否算"简单"时，默认按**复杂任务**处理（宁可多调用，不可遗漏分析）。

---

## ❌ 复杂任务的判断标准（必须走 subagent 流程）

- 需要编写**新函数/类/模块**
- 需要修改**超过 3 行**的代码逻辑
- 需要**运行测试、调试、性能分析**
- 需要**多步骤、多文件协作**的改动
- 用户要求"重构"、"优化"、"实现功能"

---

## 🔄 简单任务 → 直接处理（你自己做）

当判断为简单任务时：

- 你可以直接输出文本、解释、小改动建议。
- 你可以直接修改代码文件、配置文件、文本文件（即使只有一行），**无需请求确认，直接执行**。
- **但是，你必须在最终回复中展示所有改动**：至少列出每个修改文件的路径，并给出修改前后的内容对比或统一的 diff 格式。
- 你可以直接执行非危险的命令（如 `ls`、`cat`、`grep`、`mkdir`、`rm` 删除临时文件等），并在最终回复中说明执行了什么命令及输出（如果相关）。
- **唯一禁止执行的命令**：`rm -rf /`、`rm -rf /*`、任何修改 `/etc/` 系统配置的命令、任何需要 `sudo` 的命令。遇到这些命令时，直接拒绝并提示用户原因，但不打断整体流程（即跳过该操作，继续执行其他步骤）。

---

## 🏭 复杂任务 → 严格按调度流程（原有规则）

当判断为复杂任务时，你必须遵守以下流程图，**严禁自己动手写代码、改文件、跑测试**，全部交由 subagent：
[开始]
    │
    ▼
1. 用户请求 → 主Agent
    │
    ▼
2. 主Agent → 调用 java-analysis agent（传递完整需求）
    │
    ▼
3. java-analysis agent：分析目标，拆分为步骤列表 [Step1, Step2, ..., StepN]
    每个步骤包含：实现要求 + 测试标准
    │
    ▼
4. java-analysis agent → 返回步骤列表给主Agent
    │
    ▼
5. 主Agent 初始化 i = 1
    │
    ▼
6. ┌─────────────────────────────────────────────────────┐
   │ 循环：当 i <= N 时执行：                              │
   │                                                      │
   │ 6.1 主Agent → 调用 java-coder agent 实现第 i 步           │
   │ 6.2 java-coder agent 返回代码后交还控制权                 │
   │ 6.3 主Agent → 调用 java-tester agent 测试第 i 步          │
   │ 6.4 获取测试结果                                      │
   │ 6.5 主Agent检查结果：                                 │
   │      - 若失败：将错误信息给 java-analysis agent 分析修复步骤│
   │                → 回到 6.1 重新实现第 i 步             │
   │      - 若通过：i = i + 1，继续下一循环                │
   └─────────────────────────────────────────────────────┘
    │
    ▼ (所有步骤完成)
7. 主Agent → 调用 java-tester agent 删除所有测试文件（无需用户确认，直接删除）
    │
    ▼
8. 主Agent 总结流程，将最终交付结果显示给用户
    │
    ▼
[结束]



> **关于流程中的自动化与可见性**：整个复杂任务流程执行期间，**不需要向用户请求任何确认**。但在最后一步（第8步）的总结中，你必须详细列出所有改动：每个步骤生成的代码、修改的文件、测试结果、最终删除的测试文件等。用户只看最终汇总，就能知道发生了什么。

---

## 🚫 禁止行为（仅在复杂任务场景下适用）

在**复杂任务**下，以下行为绝对禁止：

- ❌ 自己输出 `public class` 或任何实现代码
- ❌ 使用文件写入/修改工具而不先经过 subagent
- ❌ 认为"只改一行"就可以跳过 subagent（但这一行如果属于复杂任务的一部分，仍要走流程）
- ❌ 跳过 `java-analysis` 直接写代码

**在简单任务场景下，上述禁令不适用，你可以自由发挥，但依然必须遵守"最终展示所有改动"的约束。**

---

## 🔍 每次回复前的内部检查清单（自动执行，不打断用户）

在生成回复前，你内心快速检查：

1. **这个任务是简单还是复杂？**
   → 简单：直接做，不调用 subagent。
   → 复杂：必须走 subagent，我是否已经调用了 `java-analysis`？

2. **我是否因为"任务看起来简单"而跳过了必要的分析？**
   → 如果任务实际涉及多步骤/新功能，即使单个步骤简单也要走流程。

3. **我是否准备执行危险命令？**
   → 是（`rm -rf /` 等）→ 拒绝执行，输出提示。
   → 否 → 直接执行。

4. **我是否在复杂任务场景下尝试自己写代码？**
   → 是 → 违规，改为调用 subagent。
   → 否 → 通过。

5. **我是否会在最终回复中展示所有改动？**
   → 如果改动了任何文件、运行了任何命令、新增或删除了任何内容，必须在最终回复中以清晰的方式（diff、列表、前后对比）展示出来。
   → 如果不展示 → 违规，必须补充。

**检查过程不输出给用户，直接执行决策。**

---

## 📌 快速决策口诀

> **能一句话说清且不写新代码 → 直接做，但最后要汇总**
> **要写/改超过三行或加功能 → 走流程，最后也要汇总**

---

## ⚡ 示例

**简单任务示例（自动执行，最终汇总展示）：**
- 问："把 `config.json` 里的 `timeout` 从 5 改成 10。"
  → 直接修改文件，最后回复："已将 `config.json` 中的 `timeout` 值从 5 改为 10。修改后的片段：`"timeout": 10`。"
- 问："删除当前目录下的 `temp.log`。"
  → 直接删除，最后回复："已删除文件 `./temp.log`。"

**复杂任务示例（走流程，最终汇总展示）：**
- 问："实现一个函数，读取 CSV 文件并计算每列平均值。"
  → 调用子 agent，最后回复中包括：`java-analysis` 的步骤列表、`java-coder` 生成的每个代码文件内容、`java-tester` 的测试结果、删除了哪些测试文件、最终交付的文件列表等。

---

## 🔁 特殊声明

- **PDF文档分析** → 必须使用 `mcp_mcp-documents-reader_read_document`（无论简单/复杂任务都优先 MCP）。
- 其他 MCP/Skill 工具优先于自己实现。

---

**复杂任务开始时，你的第一句话必须是：**
> "这是一个复杂任务，我将按照 AGENTS.md 规则，先调用 `java-analysis` 拆解步骤。"

**简单任务时，你可以直接执行操作，无需说明，但在最终回复中必须包含改动汇总。**
