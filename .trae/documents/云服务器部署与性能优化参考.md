# 云服务器部署与性能优化参考

> 本文档记录 game-log-service 项目在华为云轻量服务器上的部署配置和性能调优经验，供后续部署和维护参考。

## 二、性能评估

### 2.1 与本地开发机对比

```
本地机 (i7-13700KF + 64GB RAM + Samsung SSD 980 PRO)
  ┌──────────────────────────────────────────────────┐
  │  innodb_flush=2 优化后：                          │
  │  压测 3000 TPS 输入 → 写入 ~2700 TPS             │
  │  队列缓慢积压（~300条/1.1s）                       │
  │  MySQL 配置出厂默认（buffer pool 仅 128M）         │
  └──────────────────────────────────────────────────┘
                            │
                            ▼
云服务器 (2C2G + 普通云盘)
  ┌──────────────────────────────────────────────────┐
  │  预计稳定承载：300~500 TPS                        │
  │  磁盘 IOPS 是最终瓶颈（~1000-2000）               │
  │  Java 进程 + MySQL 抢 2GB 内存                    │
  └──────────────────────────────────────────────────┘
```

### 2.2 关键结论

- **瓶颈不在 CPU，在磁盘 IO**：普通云盘的 IOPS 上限远低于本地 NVMe SSD
- **内存是硬约束**：总共 2GB，Java 进程占 ~500MB，OS 占 ~300MB，留给 MySQL 的只有 ~800MB
- **3000 TPS 想都不要想**，这台机器的实际上限在 300~500 TPS

## 三、应用层配置（Spring Boot）

### 3.1 一键切换：Profile 机制

`application.yml` 已内置 Profile 支持，通过 `SPRING_PROFILES_ACTIVE` 环境变量一键切换：

| 模式 | 配置文件 | 效果 | 激活方式 |
|------|---------|------|---------|
| **云模式（默认）** | `application.yml` | batch=500, pool=3, interval=200 | 不设置任何 profile |
| **本地模式** | `application.yml` + `application-local.yml` | batch=3000, pool=10, interval=500 | `SPRING_PROFILES_ACTIVE=local` |

```bash
# 云上部署（默认就是云模式，无需额外参数）
java -jar app.jar

# 本地开发（激活 local profile）
java -jar app.jar --spring.profiles.active=local
# 或
export SPRING_PROFILES_ACTIVE=local
java -jar app.jar
```

### 3.2 application-local.yml（本地覆盖）

```yaml
# 异步写入配置 — 本地高性能（大 batch + 快刷写）
async:
  writer:
    batch-size: ${ASYNC_BATCH_SIZE:3000}
    flush-interval-ms: ${ASYNC_FLUSH_INTERVAL_MS:500}

# HikariCP 连接池 — 本地大内存，连接数可以多一些
spring:
  datasource:
    hikari:
      maximum-pool-size: ${HIKARI_MAX_POOL_SIZE:10}
      minimum-idle: ${HIKARI_MIN_IDLE:2}
```

### 3.3 参数调整说明

| 参数 | 本地值 | 云环境值 | 原因 |
|------|-------|---------|------|
| `ASYNC_BATCH_SIZE` | 3000 | **500** | 云盘慢，3000 一条要好几秒，队列瞬间满；500 条更快完成，队列不易堆积 |
| `ASYNC_FLUSH_INTERVAL_MS` | 500 | **200** | 更频繁地 flush，避免队列积压太快（500ms 空转太久） |
| `ASYNC_QUEUE_CAPACITY` | 20000 | **20000**（不动） | 保持大缓冲区，给请求层留缓冲 |
| `HIKARI_MAX_POOL_SIZE` | 10 | **3** | 只有 1 个 flush 线程写入，3 个连接足够；省内存 |
| `JAVA_OPTS` (-Xmx) | 未限制 | **-Xmx512m** | 限制 Java 堆大小，留内存给 OS 和 MySQL |

## 四、MySQL 配置（云服务器 my.cnf）

### 4.1 完整推荐配置

```ini
[mysqld]
# 内存
innodb_buffer_pool_size = 256M        # 上限了，总共才 2G 内存
innodb_log_buffer_size = 16M          # 不动，默认值够用

# 磁盘 IO
innodb_io_capacity = 500              # 普通云盘，别设太高
innodb_io_capacity_max = 1000

# 事务提交（关键性能优化）
innodb_flush_log_at_trx_commit = 2    # 最多丢 1 秒数据，业务已有文件备份兜底

# 连接数
max_connections = 20                   # 应用只用了 3 个连接，足够

# 其他
skip-name-resolve                     # 跳过 DNS 反向解析，省 CPU
innodb_flush_method = O_DIRECT        # 跳过 OS 页缓存，减少双缓冲（可选）
```

### 4.2 为什么 innodb_flush_log_at_trx_commit=2 是安全的

```
┌───────────────────────────────────────────────────────────────┐
│  数据安全保障体系（三层兜底）                                    │
│                                                               │
│  第 1 层：Log4j2 文件备份                                       │
│    submit() 中先写 logs/data/game-log.jsonl                    │
│    即使 MySQL 挂了，数据已在磁盘上                               │
│                                                               │
│  第 2 层：内存队列 + flush 线程批量入库                           │
│    正常运行时数据从队列 → JDBC batch 写入 MySQL                   │
│                                                               │
│  第 3 层：DataRecoveryRunner 启动恢复                           │
│    服务重启时扫描 jsonl 文件，与数据库对比，恢复未入库的数据       │
│                                                               │
│  结论：MySQL 级别的 ACID 对我们来说是过度保障                       │
│  牺牲最多 1 秒的数据（innodb_flush=2），换取数倍的写入性能          │
└───────────────────────────────────────────────────────────────┘
```

## 五、压测建议策略

### 5.1 分级压测方案

不要一上来就 3000 TPS，从低到高慢慢摸底：

| 轮次 | 输入 TPS | 预期写入 TPS | 观察点 |
|------|---------|-------------|--------|
| 第 1 轮 | **300** | ~300 | 队列是否稳定不积压 |
| 第 2 轮 | **500** | ~450-500 | 到达预期上限附近 |
| 第 3 轮 | **800** | ~500-600 | 摸天花板（云盘 IO 瓶颈） |

### 5.2 告警阈值

- 队列积压 > 80%（> 16000）：触发 [PRESSURE] 告警日志
- 持续 30 秒队列满：考虑降级或扩容

## 六、常见问题

### Q1：为什么本地 SSD 跑 3000 TPS，云上只能跑 300~500 TPS？

**因为磁盘 IO 是最终瓶颈。** 对比：

```
本地 Samsung SSD 980 PRO：随机写入 ~50,000 IOPS
云上普通云盘（HDD 后端）： 随机写入 ~1,000~2,000 IOPS
```

数量级差 **25~50 倍**，TPS 差 5~9 倍是正常的。

### Q2：MySQL 配置全默认不行吗？

不行。`innodb_buffer_pool_size=128M`（出厂默认）意味着 MySQL 只用了 128MB 缓存，频繁磁盘换页。但你只有 2G 内存，最多也只能给到 256M。

### Q3：为什么要把 batch-size 从 3000 降到 500？

**核心逻辑**：在 MySQL 端，单批写入耗时 ≈ 数据传输 + commit fsync。云盘 fsync 就很慢（几十到几百毫秒），3000 条一趟要 2-3 秒，队列会以 3000 TPS - 1000 TPS = 2000 TPS 的速度暴增。500 条一趟 ~500ms，队列增长慢得多。

### Q4：本地改的 my.ini（innodb_flush=2）要同步到云上吗？

**要**。本地 `my.ini` 已经持久化了 `innodb_flush_log_at_trx_commit=2`，但云上 MySQL 用的是自己的 `my.cnf`，默认是 `=1`。需要在云上也改。

## 七、部署检查清单

### 部署前

- [ ] 确认 `application.yml` 默认值已是云模式（batch=500, pool=3, interval=200）✅（已内置）
- [ ] 确认 **没有** 设置 `SPRING_PROFILES_ACTIVE=local`（云上不需要 local profile）
- [ ] 压测工具 `StressTest.java` 中 `CLOUD_MODE` 改为 `true`
- [ ] 云上 my.cnf 修改 `innodb_flush_log_at_trx_commit=2`
- [ ] 云上 my.cnf 修改 `innodb_buffer_pool_size=256M`
- [ ] 云上 my.cnf 添加 `innodb_io_capacity=500`
- [ ] 云上 my.cnf 添加 `skip-name-resolve`
- [ ] 云上 my.cnf 修改 `max_connections=20`
- [ ] 重启 MySQL 使配置生效

### 部署后

- [ ] 启动应用，检查日志无报错
- [ ] 用 300 TPS 压测 5 分钟，观察队列是否稳定
- [ ] 逐级提升 TPS 找到该配置下的上限
- [ ] 确认 `DataRecoveryRunner` 启动后能正确恢复数据

## 八、升级路径

如果未来 TPS 需求超过 500，有以下选择（按性价比排序）：

1. **升级云盘为 SSD**（性价比最高）：费用增加不大，IOPS 从 ~1000 提升到 ~10000，TPS 有望到 2000+
2. **升级实例到 4G 内存**：`buffer_pool` 可给到 1G，减少内存瓶颈
3. **应用层加反压**：队列满时返回 503，而不是降级到文件（需要客户端配合重试）
