## 二、阶段 1：请求入口与网关日志

### 1.1 Filter 注册（最高优先级）

[WebConfig.java:20-27](../src/main/java/com/gamelog/config/WebConfig.java)
```java
@Bean
public FilterRegistrationBean<ContentCachingFilter> contentCachingFilter() {
    FilterRegistrationBean<ContentCachingFilter> registration = new FilterRegistrationBean<>();
    registration.setFilter(new ContentCachingFilter());
    registration.addUrlPatterns("/*");
    registration.setName("contentCachingFilter");
    registration.setOrder(Ordered.HIGHEST_PRECEDENCE);  // 最早执行
    return registration;
}
```

### 1.2 Filter 行为

| 步骤 | 操作 |
|------|------|
| 1 | 包装 `request/response` 为 `ContentCachingRequestWrapper/ResponseWrapper` |
| 2 | 记录 `startTime` |
| 3 | 调用 `filterChain.doFilter()` 放行请求到 Controller |
| 4 | finally 块中：记录耗时 + 写日志 + `responseWrapper.copyBodyToResponse()`（**必须调用**否则客户端收不到响应） |

### 1.3 日志内容

| 路径 | 日志内容 |
|------|---------|
| 任何路径 | 请求行：`>>> GET /api/game-logs?xxx \| ip=192.168.1.1 \| status=200 \| 12ms` |
| `/api/**` | + 请求体：`>>> REQUEST BODY: {...}` |
| `/api/**` | + 响应体：`<<< RESPONSE BODY: {...}` |
| 静态资源 | 仅请求行（避免乱码） |

### 1.4 写到哪里

- **Logger 名称**：`"GatewayLogger"`
- **Appender**：`RollingFile` name="GatewayLog"
- **输出文件**：`logs/gateway/gateway.log`（按天滚动，保留 7 天）

### 1.5 关键设计点

- ⚠️ **绕开了 SLF4J**：Filter 直接使用 `LogManager.getLogger("GatewayLogger")`（Log4j2 原生 API），不依赖 SLF4J 桥接
- ⚠️ **`@Order(HIGHEST_PRECEDENCE)`**：确保在所有 Filter 之前执行
- ⚠️ **body 最大 2000 字符**（`MAX_BODY_LENGTH`）：防止大 body 撑爆日志

## 四、阶段 3：Service 层

### 4.1 写入路径

```
Controller.createGameLog(dto)
  → Service.createGameLog(dto)
    → DTO → GameLog Entity 转换
      → gameLogAsyncWriter.submit(gameLog)  ← 进入异步链
```

### 4.2 缓存配置

```java
Caffeine.<Object, Object>newBuilder()
    .expireAfterWrite(10, TimeUnit.SECONDS)  // 写后 10 秒过期
    .maximumSize(5000)                        // 最大 5000 条
    .recordStats()
```

| 配置项 | 值 | 说明 |
|--------|-----|------|
| 缓存名 | `stats` | 所有统计接口共享 |
| 过期策略 | write 后 10 秒 | 不是访问后 10 秒（TTL vs TTl） |
| 最大容量 | 5000 | 远超实际 key 数（8 个） |

### 4.3 8 个独立统计接口（按执行耗时排序）

| # | 接口 | 涉及 SQL | 索引 |
|---|------|---------|------|
| 1 | today-count | `COUNT(*) WHERE created_at BETWEEN ?` | idx_created_at |
| 2 | avg-duration | `AVG(duration) WHERE created_at BETWEEN ?` | idx_created_at |
| 3 | recent-logs | `findTop10 OrderBy createdAt DESC` | idx_created_at |
| 4 | trend | `GROUP BY DATE(created_at)` | idx_created_at |
| 5 | game-distribution | `GROUP BY game_name WHERE created_at BETWEEN ?` | **idx_game_name_created** |
| 6 | action-distribution | `GROUP BY action WHERE created_at BETWEEN ?` | **idx_action_created** |
| 7 | player-leaderboard | `GROUP BY player WHERE created_at BETWEEN ?` | **idx_player_created** |
| 8 | hourly-activity | `GROUP BY DATE, HOUR(created_at)` | idx_created_at |

> 文档说："200w 数据的统计查询从 30+ 秒降至 < 1 秒"，靠的就是 V1.0.1 复合索引。

## 五、阶段 4：异步写入 ⭐ 核心

### 5.1 完整写入链路

```
GameLogService.createGameLog(dto)
  → gameLogAsyncWriter.submit(gameLog)
    ├─ ① dataLogWriter.logData(gameLog)        ← 写文件备份，同步阻塞
    │    └─ objectMapper.writeValueAsString()    ← JSON 序列化
    │       └─ DataLogger.info(json)            ← Log4j2 同步写入（ReentrantLock 保护）
    │          └─ RollingFile → game-log.jsonl  ← 落到磁盘
    │    └─ 返回 boolean (true/false)
    │         └─ false 时 → log.warn("[FILE] 日志文件写入失败，数据仅靠队列保障")
    │
    ├─ ② queue.offer(gameLog)                   ← 进入内存队列
    │    └─ flush 线程（gamelog-flush）：
    │         ├─ queue.poll() 取首条
    │         ├─ queue.drainTo(batch, batchSize) 批量取
    │         └─ gameLogRepository.saveAll(batch) 批量入库（TABLE + batch 真生效）
    │
    └─ 返回 boolean (offered/降级结果)
         └─ false → queue.offer() 已降级同步 save(gameLog)
```

### 5.2 关键配置（[AsyncConfig.java](../src/main/java/com/gamelog/config/AsyncConfig.java)）

| 配置项 | 默认值 | 作用 |
|--------|--------|------|
| `queue-capacity` | 20000 | 内存队列容量 |
| `batch-size` | 200 | 批量入库大小 |
| `flush-interval-ms` | 500 | 队列空时睡眠时间 |

> ✅ **已修复**：原先还配了 `core-pool-size`（4）和 `max-pool-size`（8）对应一个 `ThreadPoolTaskExecutor` Bean + `@EnableAsync`，但 `GameLogAsyncWriter` 从未使用它们（用的是**自建的 `LinkedBlockingQueue` + 单线程 flush**）。现已清理，AsyncConfig 只保留 3 个真实使用的配置。

### 5.3 双重保障机制

| 保障点 | 时机 | 落点 |
|--------|------|------|
| **第 1 重**：写文件 | `submit()` 中 `dataLogWriter.logData()` | `logs/data/game-log.jsonl`（按天滚动） |
| **第 2 重**：入队列+批量入库 | flush 线程轮询 | MySQL `game_log` 表 |
| **降级**：同步入库 | 队列满时 | `gameLogRepository.save()` |

> **补充**：`logData()` 返回 `boolean`，失败时 `submit()` 打 `[FILE]` WARN 日志（不阻塞流程，数据仍通过队列保障）。

### 5.4 关闭钩子

```java
Runtime.getRuntime().addShutdownHook(new Thread(() -> {
    queue.drainTo(remaining);             // 取出队列剩余
    gameLogRepository.saveAll(remaining);  // 同步入库
}, "gamelog-shutdown-hook"));
```

> **注意**：关闭钩子中**不需要**再写文件备份，因为每条数据在 `submit()` 时（第 71 行 `logData()`）已先于 `offer()` 写到 `game-log.jsonl`，文件备份已存在。

> ⚠️ **风险点**：钩子依赖正常退出。如果 `kill -9`（SIGKILL）强制杀进程，钩子不会执行，内存队列里的数据会全部丢失——这就是为什么需要 DataRecoveryRunner。

## 六、阶段 5：停服与重启恢复 ⭐ 关键

### 6.1 触发时机

`ApplicationRunner.run()` 在 Spring 容器**完全启动后**、**接收请求前**执行一次。

### 6.2 恢复流程

```
Spring Boot 启动完成
  ↓
DataRecoveryRunner.run() 被调用
  ↓
① 检查 logs/data/ 目录是否存在
  ↓
② 扫描最近 3 天的日志文件：
   - game-log.jsonl（当前文件）
   - game-log-YYYY-MM-DD.jsonl（滚动文件）
  ↓
③ 逐行 readValue(line, GameLog.class) 解析为 GameLog
   - 跳过空行
   - 解析失败的行数累加（最多 debug 打印前 3 行）
  ↓
④ 调用 getExistingKeys() 获取"数据库中已存在的 key"
   - ✅ 已修复：按 playTime 范围一次查 DB，构建已存在 key 集合
  ↓
⑤ gameLogRepository.saveAll(toRecover) 批量入库
   - 若 saveAll 抛异常 → 返回 -1 → 不清空文件，下次启动重试
  ↓
⑥ 清理 RETENTION_DAYS(7) 之前的旧日志文件
  ↓
⑦ 所有文件恢复成功后 → 截断 game-log.jsonl（避免文件无限增大）
   - 若任一文件恢复失败（saveAll 抛异常）→ 不清空，下次启动重试
```

### 6.3 ✅ 唯一键定义与约束

数据恢复时用以下 4 元组进行查重：

```
gameName | player | action | playTime
```

> ✅ **已修复**：
> - **P0-1**：`getExistingKeys()` 已重写，按 `playTime` 范围一次查 DB，构建已存在 key 集合（不再假实现）
> - **P0-2**：方法参数未使用的 bug 自然解决
> - **P0-3**：数据库已通过 [V1.0.2__add_unique_constraint.sql](../db/migration/V1.0.2__add_unique_constraint.sql) 添加 `(game_name, player, action, play_time)` 唯一约束，重复入库会被 MySQL 拒绝

### 6.4 清理与截断

#### 清理过期文件
- 保留 `RETENTION_DAYS = 7` 天的文件
- 解析文件名中的日期（`game-log-2026-06-08.jsonl`）
- 删除早于 7 天前的文件
- 当前文件 `game-log.jsonl`（无日期）**不会被删除**（不含 `-`）

#### 同步后截断当前文件
- 所有文件恢复成功后，调用 `truncateCurrentFile()` 将 `game-log.jsonl` 截断为 0 字节
- 若任一文件恢复失败（`saveAll` 抛异常），**不清空**，下次启动重试
- 目的：避免日志文件无限增大，保证启动恢复时性能稳定

## 七、完整时序图

### 7.1 正常请求 → 入库时序

```
[客户端]  POST /api/game-logs
   │
   ▼
[ContentCachingFilter]  T0 记录开始时间
   │
   ▼
[GameLogController]  T1 接收请求，DTO 校验
   │
   ▼
[GameLogService]  T2 业务处理
   │
   ▼
[GameLogAsyncWriter.submit()]
   ├─→ [DataLogWriter.logData()]  T3 序列化 → log → 落盘 logs/data/game-log.jsonl
   └─→ [queue.offer()]  T4 进入内存队列
   │
   ▼
[GameLogController]  T5 返回 202 ACCEPTED
   │
   ▼
[ContentCachingFilter]  T6 finally 写 gateway.log + copyBodyToResponse
   │
   ▼ (同时，flush 线程在跑)
[gamelog-flush 线程]  T7 轮询队列 → drainTo batch
   │
   ▼
[GameLogRepository.saveAll]  T8 批量写 MySQL (HikariCP)
```

**关键时延**：
- T5（返回 202）：毫秒级，**不等数据库**
- T8（实际入库）：500ms 后批量执行（flush-interval）

### 7.2 崩溃 → 重启时序

```
[T1] 服务运行中，队列里还有 5000 条数据
[T2] kill -9 强制杀进程 ← 内存数据丢失！
[T3] 重启服务
   ↓
[T4] Spring 启动
[T5] DataRecoveryRunner.run() 触发
   ↓
[T6] 扫描 logs/data/game-log-*.jsonl
[T7] 解析每行 → logsFromFile（包含崩溃前 5000 条）
[T8] getExistingKeys() ← ⚠️ Bug：返回的就是 logsFromFile 的 key
[T9] toRecover = logsFromFile - 实际数据库key = logsFromFile（去重失效）
[T10] saveAll(toRecover)  ← 5000 条重新入库（与原有数据重复）
[T11] 服务开始接收请求
```

**问题**：
- ✅ 文件备份有效：5000 条不会丢
- ❌ 数据库会重复 1 遍（如果服务反复启停，会膨胀 N 倍）

## 十一、扩展方案：双源同步（gateway.log + game-log.jsonl）

### 11.1 核心洞察

**`gateway.log` 才是"用户日志存在的最有力证明"**。理由：

- 它是所有 HTTP 请求的入口记录，由 [ContentCachingFilter.java](file:///e:/trae_Projects/game_log/src/main/java/com/gamelog/config/ContentCachingFilter.java) 在所有业务逻辑之前记录
- 它包含**完整请求体**（即 [GameLogCreateDTO](file:///e:/trae_Projects/game_log/src/main/java/com/gamelog/dto/GameLogCreateDTO.java) 的全部字段）
- 即使 `DataLogWriter` 整个挂掉（不仅是我们修复的 Jackson 序列化失败，还包括未来任何未知故障），`gateway.log` 仍忠实记录
- 它还包含 `gateway.log` 独有的 `IP / status / duration` 等运维信息

### 11.2 两个恢复源的对比

| 维度 | `logs/gateway/gateway-*.log` | `logs/data/game-log-*.jsonl` |
|------|----------------------------|------------------------------|
| **写入时机** | 请求进入时（最早） | `DataLogWriter.logData()` 业务处理时 |
| **记录内容** | HTTP method/URI/IP/status/duration + 完整请求体 | GameLog JSON（gameName/player/action/detail/playTime/duration） |
| **抗绕过能力** | ✅ 即使 DataLogWriter 整个挂掉也记录 | ❌ 可能丢失（已被 Jackson Bug 验证过） |
| **解析难度** | ❌ 复杂（log 前缀 + JSON 混合） | ✅ JSONL 容易 |
| **包含 IP** | ✅ | ❌ |
| **包含 status/duration** | ✅ | ❌ |
| **包含 createdAt** | ❌（但有日志时间戳 `yyyy-MM-dd HH:mm:ss.SSS`） | ❌（由 `@PrePersist` 生成） |
| **文件大小（4000 TPS / 1 天）** | 约 1-2 GB（每条 300-500 字节含前后文） | 约 64 GB（每条约 200 字节） |
| **证据强度** | ⭐⭐⭐⭐⭐ 不可绕过 | ⭐⭐⭐ 业务过滤后 |

### 11.3 方案对比

#### 方案 A：双源融合 + 真正去重（推荐）⭐

**核心思路**：两个源都解析，**内存 Set 去重**后入库。

```
DataRecoveryRunner.run()
  ↓
① 扫描所有恢复源
  ↓
② 解析 → List<GameLog>（去重）
  ├─ game-log.jsonl（优先，解析快）→ Set<key> 去重
  └─ gateway-*.log（补充，证据强）→ Set<key> 去重
  ↓
③ 真正查询数据库 → 过滤出真正未入库的
  ↓
④ saveAll()（依赖 UNIQUE 约束兜底）
  ↓
⑤ 清理两个目录的过期文件
```

**优点**：
- ✅ 即使 JSONL 漏写一条，gateway.log 仍能恢复
- ✅ 解析简单 + 证据强 = 双保险
- ✅ 内存 Set 去重避免内存级重复入库
- ✅ 真正数据库查重（依赖 P0 修复后的 getExistingKeys）

**缺点**：
- ❌ 启动时扫描两个目录，耗时增加
- ❌ gateway.log 解析器需要稳定（格式变动会失败）

#### 方案 B：gateway.log 优先 + JSONL 兜底

**核心思路**：以 gateway.log 为唯一可信源，JSONL 作为解析失败时的备选。

**优点**：单一可信源
**缺点**：gateway.log 解析失败会丢失所有数据

#### 方案 C：双源独立处理 + 交叉验证

**核心思路**：两个源各自独立解析、独立查重入库，最后生成差异报告。

**优点**：可发现两个源的不一致（监控价值）
**缺点**：复杂，差异报告需要额外工作

#### 11.4.2 解析器（伪代码）

```java
public class GatewayLogParser {
    
    private static final Pattern REQUEST_BODY_PATTERN = 
        Pattern.compile("^(\\d{4}-\\d{2}-\\d{2} \\d{2}:\\d{2}:\\d{2}\\.\\d{3}).*>>> REQUEST BODY: (.+)$");
    
    private static final DateTimeFormatter LOG_TIMESTAMP = 
        DateTimeFormatter.ofPattern("yyyy-MM-dd HH:mm:ss.SSS");
    
    public GameLog parse(String line) {
        Matcher m = REQUEST_BODY_PATTERN.matcher(line);
        if (!m.matches()) return null;
        
        // 1. 提取时间戳
        LocalDateTime createdAt = LocalDateTime.parse(m.group(1), LOG_TIMESTAMP);
        
        // 2. 提取 JSON
        String json = m.group(2);
        
        // 3. 反序列化为 GameLogCreateDTO
        GameLogCreateDTO dto = objectMapper.readValue(json, GameLogCreateDTO.class);
        
        // 4. 转换为 GameLog
        GameLog log = new GameLog();
        BeanUtils.copyProperties(dto, log);
        log.setCreatedAt(createdAt);  // 用日志时间戳，更精确（毫秒级）
        return log;
    }
}
```

#### 11.4.3 合并去重（伪代码）

```java
public void run(ApplicationArguments args) {
    Set<String> seenKeys = new HashSet<>();           // 内存级去重
    List<GameLog> allLogs = new ArrayList<>();
    
    // ① 解析 JSONL（快、简单）
    for (Path file : getRecentJsonlFiles()) {
        for (GameLog log : parseJsonlFile(file)) {
            String key = buildKey(log);
            if (seenKeys.add(key)) {  // add 返回 true 表示新增成功
                allLogs.add(log);
            }
        }
    }
    log.info("JSONL 源解析到 {} 条（去重后）", allLogs.size());
    
    // ② 解析 gateway.log（兜底）
    int beforeGateway = allLogs.size();
    for (Path file : getRecentGatewayFiles()) {
        for (GameLog log : parseGatewayFile(file)) {
            String key = buildKey(log);
            if (seenKeys.add(key)) {
                allLogs.add(log);
            }
        }
    }
    log.info("gateway 源补充 {} 条（去重后新增）", allLogs.size() - beforeGateway);
    
    // ③ 真正查数据库
    Set<String> existingKeys = queryDatabaseForExistingKeys(allLogs);
    List<GameLog> toRecover = allLogs.stream()
        .filter(log -> !existingKeys.contains(buildKey(log)))
        .collect(Collectors.toList());
    
    log.info("待恢复 {} 条（数据库未存在）", toRecover.size());
    
    // ④ 批量入库
    if (!toRecover.isEmpty()) {
        try {
            gameLogRepository.saveAll(toRecover);
            log.info("成功恢复 {} 条记录", toRecover.size());
        } catch (DataIntegrityViolationException e) {
            // 唯一约束兜底：逐条插入，跳过已存在的
            int saved = 0, skipped = 0;
            for (GameLog log : toRecover) {
                try {
                    gameLogRepository.save(log);
                    saved++;
                } catch (DataIntegrityViolationException ex) {
                    skipped++;  // 已存在，跳过
                }
            }
            log.info("逐条插入结果: 成功={}, 跳过(已存在)={}", saved, skipped);
        }
    }
    
    // ⑤ 清理过期文件
    cleanupOldFiles(JSONL_DIR, RETENTION_DAYS);
    cleanupOldFiles(GATEWAY_DIR, RETENTION_DAYS);
}
```

```java
private Set<String> queryDatabaseForExistingKeys(List<GameLog> logs) {
    // 提取时间范围
    LocalDateTime minTime = logs.stream()
        .map(GameLog::getPlayTime)
        .min(LocalDateTime::compareTo)
        .orElse(LocalDateTime.now().minusDays(7));
    LocalDateTime maxTime = logs.stream()
        .map(GameLog::getPlayTime)
        .max(LocalDateTime::compareTo)
        .orElse(LocalDateTime.now());
    
    // 批量查询数据库（避免 N+1）
    return gameLogRepository.findExistingKeysInRange(minTime, maxTime).stream()
        .map(arr -> buildKey(arr[0].toString(), arr[1].toString(), 
                              arr[2].toString(), (LocalDateTime) arr[3]))
        .collect(Collectors.toSet());
}
```

### 12.3 3 个按顺序执行的 IO 操作（同一个 Tomcat 线程上）

**⚠️ 注意：不是并发的，是严格顺序执行的。**

在一个 Tomcat 线程上的一次请求中，执行顺序如下：

```java
filterChain.doFilter()
  └─ submit(gameLog)
       ├─ Ⅰ dataLogger.info(json)       // ← 写 game-log.jsonl
       └─ Ⅱ queue.offer()               // ← 入队
finally:
  └─ Ⅲ log.info(">>> ...")             // ← 写 gateway.log
```

即 **Ⅰ → Ⅱ → Ⅲ 严格顺序**，不存在"并行写两个文件互相锁"的问题。

**Log4j2 内部锁的情况**：

| 操作 | 对应的 Appender | 文件 | 内部锁 | 锁持有时间 |
|------|----------------|------|--------|-----------|
| Ⅰ `DataLogger.info()` | DataLog | `game-log.jsonl` | DataLog 独有 `ReentrantLock`① | ~0.05ms |
| Ⅲ `GatewayLogger.info()` | GatewayLog | `gateway.log` | GatewayLog 独有 `ReentrantLock`② | ~0.05ms |

锁①和锁②**完全独立**，互不影响。真正有竞争的是"多个 Tomcat 线程同时写同一个 Appender"（4000 线程争抢锁① 或 4000 线程争抢锁②），但锁持有时间极短（~0.05ms），不会成为瓶颈。

**单次请求的总耗时**：

| 操作 | 耗时 | 是否阻塞 |
|------|------|---------|
| Ⅰ `dataLogger.info(json)` + Jackson 序列化 | ~1ms | 同步阻塞（Log4j2 同步 Appender） |
| Ⅱ `queue.offer()` | < 0.1ms | 非阻塞（仅入内存队列） |
| Ⅲ `log.info(">>> ...")` + Jackson 序列化 | ~1ms | 同步阻塞（Log4j2 同步 Appender） |
| **合计** | **< ~2ms** | — |

### 12.4 关闭时：线程竞争时序

```
[JVM shutdown 信号]
    │
    ├─ ① Spring Context 关闭（@PreDestroy）
    │     └─ GameLogAsyncWriter：无 @PreDestroy（钩子由构造器 registerShutdownHook 注册）
    │
    ├─ ② JVM 关闭钩子开始执行
    │     └─ gamelog-shutdown-hook (线程名)
    │         ├─ queue.drainTo(remaining)        ← drain 队列中剩余数据
    │         │    （数据在 submit() 时已写入 jsonl，此处不需再写文件）
    │         └─ gameLogRepository.saveAll()     ← 同步入库（可能失败）
    │
    ├─ ③ gamelog-flush 可能正在执行
    │     ├─ queue.drainTo(batch)                ← 和关闭钩子同时 drain 是安全的
    │     ├─ flushBatch(batch) → saveAll()       ← 可能正在入库
    │     └─ 守护线程 → JVM 退出时强制终止       ← 这个 saveAll 可能被中断！
    │
    └─ ④ JVM 退出
          └─ 守护线程 gamelog-flush 被强制中断
```

**并发风险**：

| 风险场景 | 后果 | 严重性 |
|---------|------|--------|
| 关闭钩子的 `saveAll()` 与 gamelog-flush 的 `saveAll()` 有**重复数据** | 数据库中部分记录被入库两次 | 🟡 低（缺 UNIQUE 约束时会重复） |
| gamelog-flush 的 `saveAll()` 在 JDBC 执行途中被强制终止 | 部分插入成功（依赖 MySQL 是否回滚） | 🟡 中（文件备份仍在） |
| 关闭钩子的 `saveAll()` 失败 | 数据留在文件，下次启动恢复 | ✅ 可接受 |

### 12.5 吞吐量分析：4000 TPS 时队列会满吗？

#### gamelog-flush 单线程吞吐计算

```java
// 一次循环的实际工作：
first = queue.poll();                     // ~0ms（非阻塞）
queue.drainTo(batch, 200 - batch.size()); // ~0ms（非阻塞，仅加锁）
saveAll(batch);                            // 主力耗时
```

**关键变量**：`flushBatch(batch)` 中 `gameLogRepository.saveAll(200)` 的耗时。

| MySQL 环境 | saveAll(200) 耗时 | 理论吞吐 |
|-----------|-----------------|---------|
| 本地 MySQL（SSD） | ~20ms | 200 / 0.02 = **10,000/s** |
| 本地 MySQL（HDD） | ~50ms | 200 / 0.05 = **4,000/s** ✅ **刚好匹配** |
| 远程 MySQL（内网） | ~80ms | 200 / 0.08 = **2,500/s** ❌ 队列会涨 |
| 远程 MySQL（公网） | ~150ms | 200 / 0.15 = **1,333/s** ❌ 队列暴涨 |

**结论**：
- **本地环境下** 4000 TPS 时理论上能跟上（÷边界线上）
- **如果 `saveAll` 耗时 > 50ms**，队列会持续增长，几秒内填满 20000 容量
- **队列满后**所有请求降级为 Tomcat 线程同步入库 → 200 个 Tomcat 线程变为同步写 DB → 性能骤降

#### 降级后的行为

```
队列满 → queue.offer(gameLog) 返回 false
         ↓
  Tomcat 线程同步 save(gameLog)     ← 阻塞 JDBC 写
         ↓
  每个请求变成 1 次 JDBC 写          ← 4000 TPS = 4000 次 INSERT
         ↓
  HikariCP 连接池耗尽（默认 10 个连接）
         ↓
  连接池等待 → Tomcat 线程阻塞 → 响应时间飙升 → 503
```

### 12.6 读路径：线程分析

```
[Tomcat 线程] http-nio-8080-exec-N

    GET /api/game-logs/stats → GameLogController
         ↓
    GameLogService.getStats()          ← or 独立接口
         ↓
    @Cacheable 检查 Redis 缓存
    ├─ [缓存命中] → 直接返回          ← 0ms（Tomcat 线程缓存读取）
    └─ [缓存未命中] → 查数据库
          ↓
       gameLogRepository.findXXX()    ← JDBC 阻塞读
          ↓
    返回 Page<GameLog> 给客户端
```

- 读路径**全部在 Tomcat 线程**上同步执行
- `@Cacheable` 通过 Redis（lettuce）缓存，Redis 是异步线程但在 Tomcat 视角是阻塞等待
- 统计接口每 10 秒失效一次（`@Cacheable(value = "stats")`）

### 12.8 线程模型总结

```
写路径：Tomcat 线程 (2 次文件 IO + 1 次入队) → gamelog-flush (批量 saveAll)
          ↓                                      ↓
     同步 + 非阻塞                           异步 + 阻塞
          ↓                                      ↓
     响应 <5ms                             吞吐 ~4000/s（本地）

读路径：Tomcat 线程 (缓存或 DB)
          ↓
     同步阻塞
```

**核心结论**：
1. ✅ **写 gateway.log 和 game-log.jsonl 是同步 IO**，在 Tomcat 线程上执行，但耗时极短（< 2ms）
2. ✅ **gamelog-flush 是单线程**，吞吐量取决于 `saveAll` 的 MySQL 性能
3. ✅ **LinkedBlockingQueue 内部线程安全**（ReentrantLock）
4. ⚠️ **关闭时守护线程被强制终止**是最大的并发风险（P1-2）
5. ⚠️ **@Async 线程池已闲置**（EventPublisher 已删除），可清理或复用
6. ❌ **当前代码无写入路径的并发 Bug**，但如果要支持更高 TPS，单线程 flush 会成为瓶颈

### 已知精度限制

`getExistingKeys()` 修复后使用 `findByPlayTimeBetween(minTime, maxTime)` 查询数据库。
由于 `playTime` 通过 JSON 序列化/反序列化时精度会丢失（纳秒截断），若 DB 中记录了含纳秒的 `playTime`，查询可能匹配不到完全一致的结果。
**影响**：极少量记录可能被误判为"未入库"而重复恢复。
**缓解**：
- 入库前调用 `.withNano(0)` 统一精度（当前代码未做，后续可改）
- 或在启动恢复时额外比较其他字段，而非完全依赖时间范围
- **P0-3** 唯一约束是最终兜底
