# 接口拆分与独立缓存方案

## Summary
将 `/api/game-logs/stats` 拆分为 8 个独立接口，前端各组件独立请求独立渲染；后端使用统一的 15 秒缓存，让单个请求都能命中缓存。配合 30 秒前端自动刷新，整体体验接近"实时"。

## Current State Analysis

### 现状问题
- `getStats()` 在后端执行 8 个聚合查询（即使 200w 数据，索引就绪后单次约 2-5 秒）
- `@Cacheable("gameLogStats", key = "'all'")` 是单一缓存 key，缓存整个对象
- 前端一个接口拿到所有数据后一次性渲染所有图表
- 用户体验：所有图表同时显示 → 没有"渐进式"加载的感觉

### 拆分收益
| 收益 | 说明 |
|------|------|
| 独立加载 | 每个图表先渲染先显示，渐进式体验 |
| 独立缓存 | 热门接口缓存命中率高 |
| 灵活扩展 | 后续新增图表不影响其他 |
| 错误隔离 | 单个接口失败不影响其他 |

## Proposed Changes

### Step 1: Service 层拆分 8 个独立方法

**文件**: `GameLogService.java`

将 `getStats()` 拆分为 8 个独立方法，每个都加 `@Cacheable`：

```java
@Cacheable(value = "stats", key = "'today-count'")
public long getTodayCount();

@Cacheable(value = "stats", key = "'trend'")
public List<Map<String, Object>> getTrend();

@Cacheable(value = "stats", key = "'game-dist'")
public List<Map<String, Object>> getGameDistribution();

@Cacheable(value = "stats", key = "'recent-logs'")
public List<GameLog> getRecentLogs();

@Cacheable(value = "stats", key = "'leaderboard'")
public List<Map<String, Object>> getPlayerLeaderboard();

@Cacheable(value = "stats", key = "'hourly'")
public List<Map<String, Object>> getHourlyActivity();

@Cacheable(value = "stats", key = "'action-dist'")
public List<Map<String, Object>> getActionDistribution();

@Cacheable(value = "stats", key = "'avg-duration'")
public double getAverageDuration();
```

### Step 2: Controller 层新增独立接口

**文件**: `GameLogController.java`

```java
@GetMapping("/stats/today-count")
public Result<Long> getTodayCount();

@GetMapping("/stats/trend")
public Result<List<Map<String, Object>>> getTrend();

@GetMapping("/stats/game-distribution")
public Result<List<Map<String, Object>>> getGameDistribution();

@GetMapping("/stats/recent-logs")
public Result<List<GameLog>> getRecentLogs();

@GetMapping("/stats/player-leaderboard")
public Result<List<Map<String, Object>>> getPlayerLeaderboard();

@GetMapping("/stats/hourly-activity")
public Result<List<Map<String, Object>>> getHourlyActivity();

@GetMapping("/stats/action-distribution")
public Result<List<Map<String, Object>>> getActionDistribution();

@GetMapping("/stats/average-duration")
public Result<Double> getAverageDuration();
```

**保留** `/api/game-logs/stats`（兼容旧版，但用 `@Deprecated` 标记）

### Step 3: 缓存配置调整

**文件**: `CacheConfig.java`

```java
// 改为统一的 15 秒缓存（所有 stats 共享）
@Bean
public CacheManager cacheManager() {
    CaffeineCacheManager manager = new CaffeineCacheManager("stats");
    manager.setCaffeine(Caffeine.newBuilder()
            .expireAfterWrite(15, TimeUnit.SECONDS)  // 可改为 10 秒
            .maximumSize(5000)
            .recordStats());
    return manager;
}
```

### Step 4: 前端独立加载

**文件**: `index.html`

重写 `loadStats()`，为每个组件独立 fetch：

```javascript
function loadStats() {
    loadTodayCount();
    loadAverageDuration();
    loadTrend();
    loadGameDistribution();
    loadRecentLogs();
    loadLeaderboard();
    loadHourlyHeatmap();
    loadActionChart();
    loadQueueStatus();  // 队列状态独立加载
}
```

每个 `loadXxx()` 函数：
- 独立 fetch（独立超时、错误处理）
- 数据回来立即渲染该组件
- 不依赖其他接口

### Step 5: 前端自动刷新改为每个组件独立刷新

```javascript
// 每个组件设置自己的定时器
setInterval(loadTodayCount, 5000);      // 5秒
setInterval(loadQueueStatus, 5000);     // 5秒
setInterval(loadTrend, 15000);          // 15秒
// ...
```

或者更简单：使用 `Promise.all` 但每个 fetch 独立超时、错误处理，让慢的接口不影响快的。

## Assumptions & Decisions

1. **接口数量**: 8 个独立 stats 接口 + 1 个 queue-status
2. **缓存策略**: 共享 `stats` 缓存区，15 秒过期，可配置
3. **保留旧接口**: `/api/game-logs/stats` 保留但标记 @Deprecated
4. **错误处理**: 单个接口失败不影响其他（前端 try-catch 隔离）
5. **自动刷新**: 30 秒全量刷新（也可改各组件独立定时器）

---

## Verification Steps

1. 启动服务，观察各组件是否独立显示
2. 按 F5 验证是否有"渐进式"渲染效果
3. 查看后端日志，确认 15 秒内只查一次数据库
4. 调整缓存时间（如 10 秒），观察响应速度
5. 测试单个接口失败时的容错性

## Cache Hit 场景示例

| 时间 | 浏览器 A 请求 | 浏览器 B 请求 | 数据库查询 |
|------|---------------|---------------|-----------|
| 0s | today-count | - | ✅ 查 |
| 1s | trend | trend | ✅ 查（不同key） |
| 5s | today-count | - | ❌ 命中缓存 |
| 16s | today-count | - | ✅ 查（已过期） |

**好处**：
- 高频接口（如 today-count）缓存命中率高
- 缓存时间可缩短到 10 秒
- 多个用户/页面共享同一份缓存
